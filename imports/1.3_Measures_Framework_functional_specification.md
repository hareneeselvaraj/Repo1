# Daily Work Management (DWM) Platform
## 1.3 Measures Framework – Functional Specification

Version: 1.0
Architecture Style: Microservices
Service Boundary: Separate Measurement Service
Scope: Core Platform – Phase 1

---

# 1. Purpose

The Measures Framework defines how the platform models, governs, links, calculates, and owns performance measures across the enterprise.

It provides the backbone for:

- operational monitoring
- daily work management boards
- abnormality detection
- review and escalation
- root cause analysis
- continuous improvement
- cross-site and cross-context reporting

This module is designed to be:

- industry-agnostic
- sector-agnostic
- configurable by tenant
- traceable to operational standards
- scalable across enterprise → region → site → department → line → shift structures

---

# 2. Business Objective

The Measures Framework must answer the following business questions:

- what should be measured?
- why is it measured?
- who owns the measure?
- where does the data come from?
- where in the process / standard does the measure apply?
- what target or threshold defines normal vs abnormal?
- how does this measure roll up to higher-level outcomes?
- how does responsibility vary by organizational context?
- what downstream objects are affected when a measure changes?

# 3. Core Design Principles

## 3.1 Single Logical Ownership

Every measure must have exactly one **Owner Position**.

That owner is a logical accountability anchor.

Resolution model:

```text
Measure
  ↓
Owner Position
  ↓
Contextual Assignment
  ↓
Responsible Person
This keeps accountability stable while allowing different responsible people by context.
________________________________________
3.2 Context-Aware Responsibility
The platform must support ownership resolution across configurable organizational contexts.
Examples:
•	Enterprise
•	Region
•	Site
•	Department
•	Line
•	Shift
•	Product Family
Large hospitals, large plants, and ESG structures often require different responsible persons by region/site/department.
________________________________________
3.3 Mandatory Operational Traceability
Every active measure must be linked to at least one business object.
Supported linkage targets:
•	Organization Unit
•	Process
•	PFC Node
•	SOP
•	SOP Step
This ensures measures are not abstract or floating metrics disconnected from operations.
________________________________________
3.4 Flexible Graph Model
The measure model must support a flexible graph, not a strict tree.
Reason:
A single operational measure may contribute to multiple parent measures.
Example:
Cycle Time
  → OEE
  → Throughput
Therefore:
•	one measure may have multiple parent relationships
•	one measure may have multiple child relationships
•	graph traversal must support lineage and impact analysis
________________________________________
3.5 Aggregation Support
Measures must support aggregation.
Supported aggregation patterns:
•	sum
•	average
•	weighted average
•	ratio
•	formula
•	custom expression
Examples:
•	Plant OEE = weighted average of line OEE
•	Enterprise carbon intensity = emissions / revenue
•	Hospital patient satisfaction = average of site scores
________________________________________
3.6 Configuration-Driven Behavior
All optional behaviors should be controlled by configuration, including:
•	enabled context dimensions
•	required target rules
•	required link rules
•	required review/approval rules
•	allowed measure types
•	allowed aggregation methods
•	required source definitions
•	whether standalone draft measures are permitted before linkage
________________________________________
4. Business Scope
The Measures Framework covers:
•	measure definition
•	measure type management
•	measure ownership
•	target and threshold management
•	source definition
•	operational linkage
•	context-aware ownership mapping
•	measure graph relationships
•	aggregation and formula configuration
•	lifecycle and governance
•	impact analysis hooks
•	search and exploration
•	downstream publishing for runtime use
It does not yet include:
•	live runtime value capture UI
•	anomaly analytics AI
•	predictive forecasting AI
•	auto-generated RCA suggestions
Those are covered in later runtime or AI phases.

5. Functional Components
5.1 Measure Registry
The system shall maintain a master registry of measures.
The registry shall allow:
•	creating measures
•	updating draft measures
•	searching measures
•	classifying measures
•	activating approved measures
•	superseding and retiring measures
Each measure shall have:
•	code
•	name
•	type
•	description
•	owner position
•	unit of measure
•	source type
•	frequency
•	criticality
•	status
•	effective dates
________________________________________
5.2 Measure Type Management
The system shall support, at minimum, the following types:
•	KPI
•	MP
•	MOP
•	CP
Definitions:
KPI
Outcome-oriented measure used to monitor strategic or operational result.
Examples:
•	OTIF
•	Patient Satisfaction
•	Carbon Emission Intensity
MP
Managing Point / process performance measure.
Examples:
•	Order Fulfillment Lead Time
•	Emergency Response Time
•	Assembly Cycle Time
MOP
Method of Operation / method performance measure.
Examples:
•	Inspection Accuracy
•	Packing Completion Time
•	Medication Accuracy
CP
Checkpoint / operational control point.
Examples:
•	Barcode Scan Confirmation
•	Drug Dose Confirmation
•	Torque Reading
The platform may optionally allow tenant-defined subtype tags such as:
•	Safety KPI
•	Financial KPI
•	ESG KPI
•	Clinical KPI
________________________________________
5.3 Measure Ownership
Every measure shall have one Owner Position.
The system shall:
•	validate owner position existence
•	resolve owner to person by context
•	expose ownership gaps
•	expose ambiguous ownership resolution
•	support large hierarchy models such as enterprise → region → site
The system shall not store multiple logical owner positions for a single measure.
Instead, multiple context-specific responsible persons shall be resolved through assignment.
________________________________________
5.4 Context Ownership Resolution
The system shall support responsible person resolution across enabled contexts.
Examples:
Manufacturing
Measure: Cycle Time
Owner Position: Production Supervisor
•	Site Chennai → Ravi Kumar
•	Site Pune → Sanjay Patil
Hospital
Measure: Emergency Response Time
Owner Position: ER Supervisor
•	Region North → Dr. Kumar
•	Region South → Dr. Rao
ESG
Measure: Energy Consumption
Owner Position: Facilities Manager
•	Region APAC → Manager A
•	Region EU → Manager B
The system shall support:
•	single-context ownership models for simpler tenants
•	multi-context ownership models for complex tenants
•	configuration-driven required context resolution
________________________________________
5.5 Measure Linkage
The system shall allow a measure to link to one or more business objects.
Supported object types:
•	Organization Unit
•	Process
•	PFC Node
•	SOP
•	SOP Step
The system shall support multiple links for the same measure.
Examples:
•	same measure linked to Process and SOP Step
•	same checkpoint linked to SOP Step and PFC Node
•	same KPI linked to Organization Unit and Process
The system shall support link scopes such as:
•	primary
•	secondary
•	reference
The system shall enforce at least one valid link before measure activation, unless tenant policy explicitly allows draft-only standalone creation.
________________________________________
5.6 Measure Graph Relationships
The system shall allow relationships between measures.
Supported relationship use cases:
•	parent-child rollup
•	contributor relationship
•	dependency relationship
•	derived-from relationship
The graph shall support:
•	one-to-many
•	many-to-one
•	many-to-many
Examples:
•	line OEE contributes to plant OEE
•	packing completion contributes to order cycle time
•	dosage verification CP contributes to medication accuracy MOP and patient safety KPI
The graph shall support lineage tracing:
•	upstream parents
•	downstream children
•	impact traversal
________________________________________
5.7 Aggregation and Calculation Model
The system shall support both direct and derived measures.
Direct Measure
Value comes from:
•	manual entry
•	external system
•	sensor/system feed
Derived Measure
Value is calculated from:
•	other measures
•	formulas
•	aggregated context values
Supported aggregation methods:
•	SUM
•	AVERAGE
•	WEIGHTED_AVERAGE
•	RATIO
•	FORMULA
•	CUSTOM_EXPRESSION
Examples:
Manufacturing
Plant OEE = weighted average(Line OEE)
Hospital
Regional Wait Time = average(Hospital Wait Time)
ESG
Carbon Intensity = Emissions / Revenue
The system shall store enough metadata for runtime and later analytics engines to evaluate formulas.
________________________________________
5.8 Targets and Thresholds
Each measure may have one or more targets with scope and validity period.
Supported target dimensions:
•	target value
•	lower control limit
•	upper control limit
•	warning threshold
•	critical threshold
Targets may vary by context.
Examples:
•	Cycle Time target differs by site
•	Patient Wait Time threshold differs by department
•	Energy Consumption target differs by plant
The system shall support effective-dated targets.
________________________________________
5.9 Source Definition
The system shall support source definitions for each measure.
Supported source types:
•	MANUAL
•	SYSTEM
•	CALCULATED
•	HYBRID
Source definition may include:
•	source system
•	source entity
•	source field
•	extraction rule
•	transformation rule
Examples:
•	MES cycle_time field
•	ERP order_completion_date
•	manual inspection result
•	calculated ratio from two published measures
________________________________________
5.10 Lifecycle and Governance
Measures shall follow a governed lifecycle.
Supported lifecycle states:
•	DRAFT
•	SUBMITTED
•	UNDER_REVIEW
•	APPROVED
•	ACTIVE / EFFECTIVE
•	SUPERSEDED
•	RETIRED
The system shall support:
•	review/approval workflows
•	versioning
•	change summaries
•	change impact hooks
•	controlled activation
________________________________________
5.11 Measure Search and Discovery
The system shall support search and filtering by:
•	measure code
•	measure name
•	type
•	owner position
•	resolved person
•	linked object
•	context
•	criticality
•	status
•	frequency
The system shall provide graph-aware exploration.
Examples:
•	find all measures linked to SOP 12
•	find all child measures contributing to OEE
•	find all measures owned by Production Supervisor
•	find all measures with ownership gaps in Site Chennai
________________________________________
5.12 Change Impact Awareness
When a measure changes, the system shall identify impacted downstream areas.
Potential impacts:
•	linked SOPs
•	linked PFC nodes
•	linked processes
•	runtime monitoring definitions
•	dashboards / DWM boards
•	abnormality thresholds
•	parent/child measure calculations
The system shall classify impact severity.

6. Functional Requirements
FS-MEA-001 Measure Creation
The system shall allow authorized users to create a new measure in Draft state.
FS-MEA-002 Unique Code
The system shall enforce tenant-scoped uniqueness of measure code.
FS-MEA-003 Measure Type
The system shall require a valid measure type.
FS-MEA-004 Owner Position
The system shall require one valid owner position.
FS-MEA-005 Context-Aware Resolution
The system shall support context-aware owner resolution through assignment.
FS-MEA-006 Linkage Support
The system shall allow linking a measure to one or more supported business objects.
FS-MEA-007 Mandatory Link Before Activation
The system shall require at least one valid measure link before activation, subject to configuration.
FS-MEA-008 Graph Relationships
The system shall allow creation of graph relationships between measures.
FS-MEA-009 Multiple Parent Support
The system shall allow a measure to have multiple parent measures.
FS-MEA-010 Multiple Child Support
The system shall allow a measure to have multiple child measures.
FS-MEA-011 Aggregation Definition
The system shall support configured aggregation methods and formula definitions.
FS-MEA-012 Target Definition
The system shall allow defining one or more effective-dated targets per measure.
FS-MEA-013 Threshold Definition
The system shall allow defining warning and critical thresholds.
FS-MEA-014 Context-Specific Targets
The system shall support target definitions scoped to enabled contexts.
FS-MEA-015 Source Definition
The system shall allow defining a measure source type and optional source metadata.
FS-MEA-016 Lifecycle Control
The system shall enforce lifecycle states and transitions.
FS-MEA-017 Review & Approval
The system shall integrate with governance workflows for submission, approval, rejection, and rework.
FS-MEA-018 Search
The system shall support measure search by code, name, type, owner, status, context, and links.
FS-MEA-019 Impact Analysis Hook
The system shall expose measure changes for downstream impact analysis.
FS-MEA-020 Ownership Gap Visibility
The system shall identify and surface ownership gaps for measures.
FS-MEA-021 Ambiguity Visibility
The system shall identify and surface ambiguous contextual owner resolution.
FS-MEA-022 Effective History
The system shall preserve historical versions and effective-date changes.
FS-MEA-023 Publish for Runtime
The system shall publish active measure definitions for downstream runtime consumption.
FS-MEA-024 Tenant Configuration
The system shall allow tenant-specific configuration of required contexts, measure rules, and aggregation options.
________________________________________
7. Business Rules
BR-MEA-001 One Logical Owner
A measure shall have exactly one owner position.
BR-MEA-002 Active Measure Needs Link
An active measure shall have at least one valid link unless explicitly exempted by configuration.
BR-MEA-003 Owner Must Exist
Owner position must exist and be active or valid per lifecycle rules.
BR-MEA-004 Activation Requires Validation
A measure cannot be activated if blocking validation errors exist.
BR-MEA-005 Graph Cycles by Policy
The system shall detect cycles in measure relationships and allow/block them based on defined policy. Recommended default: block invalid cyclic calculation dependencies.
BR-MEA-006 Target Validity
Threshold and target ranges must be logically valid.
BR-MEA-007 Context Scope Uses Enabled Dimensions
Only enabled context dimensions may be used in context-scoped ownership or target definitions.
BR-MEA-008 Approval Required
Approval is required before activation if governance policy is enabled.
BR-MEA-009 Historical Preservation
Superseded and retired measures must remain queryable historically.
BR-MEA-010 Shared Measure Allowed
A measure may be shared across multiple parents and multiple operational links.
________________________________________
8. Edge Scenarios to Support
The system must support the following scenarios.
ES-MEA-001 One Measure, Many Parents
Cycle Time contributes to both OEE and Throughput.
ES-MEA-002 One Measure, Many Links
Torque Reading linked to Process, PFC Node, and SOP Step.
ES-MEA-003 Same Owner Position, Different Sites
Production Supervisor owns the same measure across multiple sites, resolved to different people.
ES-MEA-004 Region-Level Healthcare Ownership
ER Response Time owned by ER Supervisor and resolved by Region/Hospital.
ES-MEA-005 ESG Enterprise Rollup
Region Energy Consumption rolls up to Enterprise Sustainability KPI.
ES-MEA-006 Measure With No Current Assignment
Measure has owner position but no active responsible person in a context.
ES-MEA-007 Ambiguous Assignment
Two matching assignments exist for same context with no primary rule.
ES-MEA-008 Target Change With Downstream Impact
Changing target affects board thresholds and abnormality rules later.
ES-MEA-009 Superseded Child Measure
Parent graph still references a superseded child and must warn/block depending on policy.
ES-MEA-010 Partial Context Model
Small tenant uses only Enterprise → Site, not Line/Shift.
