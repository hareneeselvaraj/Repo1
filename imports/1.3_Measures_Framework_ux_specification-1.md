# Daily Work Management (DWM) Platform
## 1.3 Measures Framework – UX Specification

Version: 1.0
Architecture Style: Microservices
Service: Measurement Service
Scope: Phase-1 Core

---

# 1. Purpose

This document defines the UX specification for the Measures Framework.

It covers:

- information architecture for measure-related screens
- navigation and page structure
- measure creation and editing experience
- graph exploration
- target and threshold management
- context-aware ownership visibility
- review and approval UX
- validation behavior
- status and lifecycle display

This document is intended for:

- product owners
- UX designers
- frontend developers
- architects
- QA teams

---

# 2. UX Design Principles

## 2.1 Measures Must Be Explainable

A user should be able to understand immediately:

- what the measure is
- why it exists
- who owns it
- where it is linked
- what target applies
- what other measures it contributes to

The UI must never make measures appear as isolated numbers.

---

## 2.2 Ownership Must Be Visible at Two Levels

Every major measure screen should display:

- Owner Position
- Responsible Person by context

Example display:

```text
Owner Position: Production Supervisor
Responsible Person: Ravi Kumar
Context: Chennai / Line 1 / Day Shift
Resolution Status: Resolved
If unresolved:
Owner Position: Production Supervisor
Responsible Person: Not Resolved
Resolution Status: Ownership Gap
________________________________________
2.3 Graph Relationships Must Be Understandable
Because the model is a flexible graph, the UI must clearly distinguish:
•	parent measures
•	child measures
•	linked operational objects
•	aggregation logic
The UI should avoid implying a strict tree if the model supports shared measures.
________________________________________
2.4 Status Must Be Obvious
All measure-related objects should display lifecycle state clearly:
•	Draft
•	Submitted
•	Under Review
•	Approved
•	Active
•	Superseded
•	Retired
Warnings and unresolved issues should also be surfaced prominently:
•	Ownership Gap
•	Ambiguous Assignment
•	Missing Target
•	Missing Link
•	Blocking Impact
________________________________________
2.5 Enterprise Context Must Be Easy to Filter
Because the platform supports enterprise → region → site and deeper structures, the UX must support:
•	context filtering
•	ownership matrix by context
•	target scope visualization
•	context chips and breadcrumb-like clarity

3. Primary User Roles and UX Behavior
3.1 Measure Author
Needs:
•	create and edit measures
•	define links
•	define targets
•	define relationships
•	submit for approval
UX preference:
•	form-heavy but structured workspace
•	fast access to related objects
•	strong validation guidance
________________________________________
3.2 Process / Standard Owner
Needs:
•	understand which measures attach to processes/PFC/SOP/steps
•	confirm traceability
•	review downstream effect of measure changes
UX preference:
•	relationship and traceability view
•	impact awareness
•	searchability by standard object
________________________________________
3.3 Reviewer / Approver
Needs:
•	review measure completeness
•	see owner and ownership gaps
•	compare targets/version changes
•	understand graph and impact quickly
UX preference:
•	concise review workspace
•	version diff
•	impact summary
•	approval action bar
________________________________________
3.4 Operations / Business Lead
Needs:
•	browse active measures
•	view ownership by context
•	understand target expectations
•	see graph lineage
UX preference:
•	read-first screens
•	filterable library
•	graph and matrix visuals
________________________________________
3.5 Admin / Governance User
Needs:
•	troubleshoot ownership gaps
•	review ambiguous resolutions
•	inspect target conflicts
•	manage configuration-related issues
UX preference:
•	workbench views
•	matrix/table screens
•	filters and bulk investigation tools
________________________________________
4. Information Architecture
4.1 Navigation Placement
Within the broader DWM app, Measures should have its own section in left navigation.
Recommended top-level nav item:
•	Measures
Secondary navigation inside Measures:
Measure Library
Measure Designer
Targets
Relationships
Graph Explorer
Ownership Matrix
Approvals
Versions
Impact Analysis
________________________________________
4.2 Object-Centric Navigation
Every measure detail page should expose tabs:
Overview | Links | Relationships | Targets | Source | Ownership | Versions | Impact | History
This ensures users do not need to switch across unrelated screens to understand one measure fully.

9. Status / Messaging Standards
9.1 Success Messages
Examples:
•	“Measure saved as Draft”
•	“Target added successfully”
•	“Measure submitted for approval”
•	“Measure activated successfully”
9.2 Warning Messages
Examples:
•	“No active target exists for some contexts”
•	“Owner resolution is unavailable for selected site”
•	“This relationship creates shared dependency; review impact”
9.3 Blocking Errors
Examples:
•	“At least one link is required before activation”
•	“Invalid cyclic dependency detected”
•	“Owner position has no valid assignment for required context”
________________________________________
10. Role-Based Access UX
10.1 Read-Only vs Edit
•	active measures may be read-only unless revision initiated
•	draft measures editable by author/editor role
•	approval actions visible only to reviewer/approver
10.2 Conditional Field Enablement
Examples:
•	target scope fields shown only if context dimensions enabled
•	formula field shown only for formula-capable aggregation methods
•	source system/entity fields shown only for SYSTEM/HYBRID source type
________________________________________
11. Enterprise Context UX Pattern
Because the product supports enterprise → region → site and beyond, every relevant measure screen should allow context awareness without clutter.
Recommended pattern:
•	top context bar for filters
•	context chips in target rows
•	ownership matrix for multi-context visibility
•	keep hidden dimensions out of UI if disabled
________________________________________
12. Key Edge Scenario UX
12.1 Shared Measure
If a measure contributes to multiple parents:
•	show badge: “Shared Measure”
•	relationships tab lists all parents
•	graph explorer highlights all parent paths
12.2 Ownership Gap
•	red or blocking indicator
•	CTA: “Open Assignment Console”
•	visible in Measure Library and Ownership tab
12.3 Ambiguous Assignment
•	amber indicator
•	show possible matching assignments if allowed
•	link to admin resolution
12.4 Missing Target
•	warning in designer
•	visible in approval review
•	optionally block activation by policy
12.5 Superseded Linked Standard
•	impact warning in Links or Impact tab
•	navigate to active linked standard version
