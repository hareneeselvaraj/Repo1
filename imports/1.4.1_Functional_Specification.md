# Daily Work Management (DWM) Platform
## 1.4 Process Standardization – Functional Specification

Version: 1.0  
Architecture Style: Microservices  
Service: Standardization Service  
Scope: Phase-1 Core (Standardization Foundation)

---

# 1. Purpose

This document defines the functional behavior of the Process Standardization module.

It covers:

- Process definition
- PFC (Process Flow Chart)
- SOP (Standard Operating Procedure)
- SOP Steps (with conditional branching)
- Ownership model (Position → Person resolution)
- Measure linkage
- Versioning and lifecycle governance
- Change control with override capability
- Yokoten (horizontal deployment)
- Future improvement traceability hooks

This module defines **how work should be done**.

---

# 2. Design Principles

## 2.1 Standardization First

The system defines:
- the expected way of performing work
- not real-time execution tracking (Phase-2)

---

## 2.2 Separation of Concerns

Clear separation:

```text
Process → PFC → SOP → SOP Steps
•	Process = business-level grouping
•	PFC = flow logic
•	SOP = detailed instruction set
•	SOP Step = atomic execution unit
________________________________________
2.3 Ownership Model (Aligned with 1.2)
•	Every Process / PFC / SOP / Step has:
o	owner_position_id
•	Responsible person is:
o	resolved dynamically via Assignment Service
•	Supports enterprise → region → site context
________________________________________
2.4 Linkage to Measures (Aligned with 1.3)
•	SOP Step ↔ Measure (primary linkage)
•	Process ↔ KPI (optional)
•	Enables:
o	traceability
o	impact analysis
o	future deviation analysis
________________________________________
2.5 Governance with Flexibility
•	Approval workflow enforced
•	Change control enforced
•	Overrides allowed with justification (no blind blocking)
•	All overrides are auditable
________________________________________
2.6 Future-Ready (PDCA Hooks)
Phase-1 does NOT implement:
•	abnormality workflows
•	PDCA lifecycle
But must support:
•	reference capture
•	traceability to future modules

•	End

________________________________________
3.3 SOP
Detailed instructions linked to PFC nodes.
Attributes
•	sop_id
•	pfc_node_id
•	sop_code
•	sop_name
•	description

•	sop_step_id

System behavior:
•	highlights impact (always)
•	allows override (configurable)
•	requires:
o	justification

•	override_flag
•	justification_text
•	approved_by
•	impacted_objects_snapshot

11. Validation Rules
11.1 Structural
•	Process must have at least one PFC
•	PFC must have start and end nodes
•	SOP must link to valid PFC node
•	SOP Step must belong to SOP
________________________________________
11.2 Governance
•	approval required before activation
•	version must be unique
________________________________________
11.3 Linkage
•	SOP Step → Measure must be valid
•	linked objects must exist
________________________________________
11.4 Override Validation
•	justification mandatory if override used
________________________________________
12. Access Control

•	Viewers → read-only
•	Admin → override governance
