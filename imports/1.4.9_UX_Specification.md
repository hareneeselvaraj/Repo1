# 1. Purpose

This document defines the UX specification for the Process Standardization module.

It covers:

- information architecture for process standardization screens
- navigation and page structure
- process authoring experience
- PFC graph design experience
- SOP and SOP step authoring
- branching and transition management
- measure linkage UX
- review / approval / activation UX
- override justification UX
- Yokoten deployment UX
- ownership and context visibility

This document is intended for:

- product owners
- UX designers
- frontend developers
- architects
- QA teams

# 2. UX Design Principles

## 2.1 Standardization Must Be Understandable

A user should be able to understand clearly:

- what the process is
- how the flow works
- what SOPs support the flow
- what each step instructs
- who owns the object
- which measures are linked
- what changed across versions

The UX must not make standards feel like disconnected documents.

---

## 2.2 Ownership Must Always Be Visible

Every major standardization screen should clearly display:

- owner position
- resolved responsible person
- context
- resolution status

Example display:

```text
Owner Position: Production Supervisor
Responsible Person: Ravi Kumar
Context: Chennai / Line 1 / Day Shift
Resolution Status: Resolved

If unresolved:

Owner Position: Production Supervisor
Responsible Person: Not Resolved
Resolution Status: Ownership Gap
2.3 PFC and SOP Must Remain Distinct

The UX must preserve clean separation:

PFC = flow logic / structure

SOP = detailed work instruction

SOP Step = atomic instruction / decision

The UI should not blur PFC into SOP or vice versa.

2.4 Graph and Branching Must Be Usable

Because both PFC and SOP steps can include branching, the UX must:

make flow direction obvious

label decision branches clearly

avoid visual clutter

allow users to inspect details without losing context

2.5 Governance Must Be Strong but Usable

The UX must support:

strict visibility of impact

optional override with justification

no silent bypass

full reviewability

The UI should make it hard to ignore impact, even when override is permitted.

2.6 Standardization Must Be Context-Aware

The product supports enterprise → region → site → department → line → shift.

The UX must:

show only enabled context dimensions

keep context display consistent

support scoped viewing and deployment

3. Primary User Roles and UX Behavior
3.1 Process Author

Needs:

create processes

build PFCs

create SOPs

define step logic

link measures

submit for approval

UX preference:

structured authoring workspace

graph canvas + form hybrid

strong validation

3.2 Functional Owner

Needs:

review process coverage

verify SOP completeness

confirm ownership and linkage

understand impact of revisions

UX preference:

object-centric views

traceability tabs

impact and version summaries

3.3 Reviewer / Approver

Needs:

review standards quickly

see changes, impacts, ownership, and linked measures

approve / reject / request changes

UX preference:

consolidated review page

compact comparison and impact display

clear action bar

3.4 Admin / Governance User

Needs:

investigate activation blockers

review override usage

monitor rollout / Yokoten adoption

trace versions and history

UX preference:

workbench-style pages

matrix and grid views

high filterability

3.5 Business / Site Lead

Needs:

browse active standards

view scoped versions for site/line

understand deployed/adopted standards

UX preference:

read-first UI

clear status and context filters

lightweight drill-downs

10. Role-Based Access UX
10.1 Read-Only vs Edit

active standards are read-only unless revision initiated

draft standards editable by author/editor roles

approval actions visible only to reviewer/approver

override actions visible only to authorized roles

10.2 Conditional Field Enablement

Examples:

override button only shown if policy allows

context scope fields shown only for enabled dimensions

transition condition fields shown only for conditional/decision transitions

11. Enterprise Context UX Pattern

All standardization screens should support context without clutter.

Recommended pattern:

top context filter bar

context chips in object headers

ownership matrices for multi-context views

hidden dimensions omitted when disabled

12. Key Edge Scenario UX
12.1 Ownership Gap

red blocking or warning indicator depending on policy

visible in libraries, detail pages, and approval review

CTA to open related ownership / assignment area

12.2 Ambiguous Assignment

amber warning

show ambiguity in ownership card or matrix

link to admin resolution flow

12.3 Broken Measure Link

warning or blocking indicator on step/process

impact banner shown before activation

12.4 Override Used

explicit override badge on version/history views

justification visible to authorized reviewers

12.5 Yokoten Deviation

adoption tracker row marked as DEVIATED

deviation note visible

does not silently disappear into completed rollout
